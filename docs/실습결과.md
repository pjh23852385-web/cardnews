## 작업 과정

### 처음에 시도한 것

첫 프롬프트는 이거였음:

> "카드뉴스자동화를 위한 업무자동화를 해볼건데
> 일단 시도하고싶은 방식에 대해서 먼저 소통하자.
>
> awesome-design-md(github.com/VoltAgent/awesome-design-md, 3일 전 올라온 따끈한 레포) 또는 design.md 어제쌤이 알려준 사이트 등 에서 스타일 하나 골라서
> ↓ design.md로 프로젝트에 넣고
> Swiper.js 카드 슬라이더 템플릿 참고해서
> ↓ Claude Code로 카드뉴스 생성
> Vercel 배포 → 링크 공유
>
> 이 방식하나가 있고,
>
> drama-agent/ 같은 오케스트레이션 형태로도 해보고 싶음. 업무 플로우를 먼저 계획해줘"

→ 시작부터 욕심 좀 있었음. "템플릿 기반 단순 자동화" vs "에이전트 오케스트레이션" 두 마리 토끼를 다 잡으려고.

Claude가 바로 코드 치는 대신 **Plan Mode**로 들어가서 같이 플랜을 짜자고 함. 결과적으로 이게 제일 큰 전환점이었음.

### 시행착오 / 어려웠던 부분

#### 1️⃣ 플랜 잡는 게 제일 오래 걸림 (에이전트 6→5→3으로 축소)

처음 계획: **6개 에이전트**
- content-planner / copywriter / slide-builder / art-director / image-generator / publisher

이게 과하다 느껴져서 점점 줄였음:

**중간 (5개)**: publisher 빼서 편집장이 배포까지 하게
**최종 (3개)**: slide-builder + art-director + image-generator 를 "아트디렉터" 한 명으로 통합

처음엔 "세분화가 더 좋을 것 같은데?" 싶었는데, 결과물 보고 확신함 — **적을수록 좋다**. 봇 3개가 그룹방에서 대화하는 게 5개보다 훨씬 보기 좋음.

플랜은 v1 → v2 → v3까지 갔고, 최종 v3 만드는 데에만 10번 이상 편집했음. 근데 그 덕분에 구현은 빨랐음.

#### 2️⃣ "의사" vs "시사점" 용어 교체 대란

중간에 이런 요청 했음:

> "의사 /의사없이 라고 적혀있는 것들 ->시사점 이라고 바꾸자"

Claude가 Grep으로 10군데 찾아서 9군데만 바꾸고 1개는 남김 (`의사소통 비용` — 맥락 다름). 꼼꼼했음.

이게 단순 단어 교체 같지만, 사실 **개념 정리**였음. "오디언스와 의사" 보다 "오디언스와 시사점"이 훨씬 명확함. 나중에 카피가 이 철학 가지고 작업하니까 결과물 퀄리티에 진짜 영향을 줌.

#### 3️⃣ 텔레그램 봇 토큰 삽질 두 번

**첫 번째 실수**: BotFather에서 봇 ID만 복사. `TELEGRAM_BOT_EDITOR=8718102331` 이렇게 10자리 숫자만. → Unauthorized

Claude가 "토큰은 `ID:해시` 형태여야 한다" 고 알려줌. 다시 복사.

**두 번째 실수**: 이번엔 토큰 앞에 숫자가 더 붙어있음.
- `88718102331:AAG...` (11자리)
- `878792408865:AAG...` (12자리)
- 표준은 10자리. → 또 Unauthorized

원인: `.env` 에 있던 기존 ID를 지우지 않고 그 뒤에 전체 토큰을 붙여넣어서 중복.

Claude가 `curl getMe` 로 3봇 각각 테스트해서 ART만 성공, EDITOR/COPY는 Unauthorized 나오는 걸 확인. 그다음 자동으로 "앞 N자리 제거하면 10자리 → 혹시 이게 맞지 않을까?" 하고 테스트해봄. **한 번에 맞췄음.** 이게 진짜 편했음.

#### 4️⃣ 봇 유저네임이 계획과 달랐음

플랜엔 `@hl_copy_bot` 이라고 써놨는데, 실제로 만든 건 `@hl_copywriter_bot`.
`@hanwha_finance_bot` 계획이었는데 실제는 `@HanwhaFinanceNewsBot`.

내가 이렇게 물었음:

> "봇 유저네임이 @hl_copywriter_bot: 이런건데 바꿔야하는거?"

Claude: "아니, 플랜 쪽을 실제 봇 유저네임에 맞춰서 업데이트하는 게 편함."

→ Grep으로 일괄 찾아서 `CLAUDE.md` / `.env.example` / 3개 에이전트 / `router.md` 한 번에 교체.

**배운 점**: 플랜 먼저 확정하고 → 봇 생성 → 플랜 갱신. 이 순서.

#### 5️⃣ 한글 텔레그램 메시지 인코딩 에러

Bash에서 curl로 JSON body에 한글 넣으니 `"strings must be encoded in UTF-8"` 에러. Windows Git Bash 환경의 특유 문제.

해결: JSON을 파일에 먼저 쓰고 `--data-binary @파일.json` 으로 전송. Unicode escape(`\uD83D`)로 이모지도 처리.

#### 6️⃣ 가장 큰 삽질 — 소스 파일을 안 읽고 시작함

`01. 카드뉴스 소스.md` 를 "AI Agent 시장 패러다임 변화" 한 편인 줄 알고 시작했음 (제목만 보고 판단).

체크포인트 ①②까지 진행하고, 카피가 본문 분석하다가 발견:

> "편집장, 이 소스 실제로는 4개 독립 주제야:
> 1. AI Agent 시장 패러다임 변화
> 2. AI 둠스데이 '2028 글로벌 지능 위기'
> 3. Aspora 성공 사례
> 4. 바이브코딩
>
> 경영진 2~3분에 4개 다 욱여넣으면 시사점이 묻혀."

→ 에스컬레이션 → 나에게 "A: 4개 통합 / B~E: 하나만 집중 / F: 시리즈" 중 선택 요청 → **A 선택** (15장 통합).

**교훈**: 편집장이 진짜로 소스를 처음부터 끝까지 다 읽고 브리핑했어야 함. 제목만 보고 짐작하면 안 됨. 이게 프롬프트 개선 포인트.

## Claude Code, 이게 진짜 좋았다

### 1️⃣ Plan Mode가 진짜 훌륭함
- 플랜 쓰는 동안 코드 못 건드리게 막아서 삽질 방지
- `ExitPlanMode` 호출할 때 내가 승인/반려 명시적으로 선택
- 중간에 "잠깐, 이 부분 다시 하자" 왔다갔다 해도 플랜 파일 하나에서 계속 갱신됨
- 승인 후 비로소 코드 작성 시작 → 이미 그림이 있어서 구현이 빠름

### 2️⃣ 병렬 WebFetch
레퍼런스 URL 4개 한 번에 읽음. 순차로 돌리면 1분 넘는 걸 15초만에.

### 3️⃣ 토큰 문제 자동 진단
봇 토큰 에러 났을 때, Claude가 `curl getMe` 3번 돌려서 "앞자리 N개 제거하면 통과" 패턴을 **한 번에** 찾음. 내가 수동으로 BotFather에 다시 들어가서 토큰 받을 필요 없었음.

### 4️⃣ 하이브리드 모드 아이디어
내가 "텔레그램에 답변해도 왜 안 읽어?" 라고 물으니까 Claude가 즉석에서 해결책 제안:
> 'Claude Code에서 명령 → 봇이 텔레그램에 중계 → 사용자는 그룹방에서 구경 + Claude Code에서 체크포인트 답변'

폴링 런타임 없이도 양쪽 장점 다 가져가는 방식. 발상이 좋았음.

### 5️⃣ 세밀한 자동 커밋
의미 있는 단계마다 Claude가 알아서 커밋. 메시지도 정성껏. 중간에 롤백하기 쉬움.

## 꿀팁 / 이렇게 하면 더 잘 됨

### 🔸 플랜에 시간 쓰는 걸 아끼지 마라
- v3까지 가는 데 플랜만 3시간. 근데 구현은 30분
- 바로 코딩부터 했으면 10번은 갈아엎었을 듯

### 🔸 API 키는 채팅에 절대 안 붙여넣기
- Claude가 매번 경고해줌. 진짜 중요함
- 대화 로그에 영구 기록돼서 유출 위험
- **`.env` 파일 직접 열어서 넣기**

### 🔸 공장 은유를 끝까지 밀고 가기
"카드뉴스 공장"이라는 메타포를 모든 곳에 일관되게 적용:
- `CLAUDE.md` = 공장 운영 매뉴얼 (안 바뀜)
- `designs/` = 디자인 카탈로그 (계속 추가)
- `design.md` = "이번 달은 이 페이지 쓰자" 포스트잇
- `sources/YYYY-MM/` = 월별 소스 투입구

나중에 새 스타일 추가 = 파일 하나, 새 편 제작 = 소스 하나, 이렇게 단순해짐.

### 🔸 페르소나에 MBTI + 실존 인물 넣기
- **편집장** = 이미경(CJ 부회장) + 강원국(청와대 연설비서관) = **ENFJ**
- **카피** = 박웅현(TBWA) + 이제석 = **INTJ**
- **아트디렉터** = 조수용(매거진B) + 배민 디자인팀 = **ENTP**

각 봇이 발화 예시 표까지 만들어두니 Telegram 메시지에서 봇들 톤이 확실히 구분됨. 그룹방에서 "아 이 말투 편집장이네" 바로 알아봄.

### 🔸 핵심 철학 한 문장 박아두기
**"오디언스와 시사점 없이 카피 안 쓴다"**

이 문장을 `CLAUDE.md` 최상단 + 각 에이전트 원칙에 넣으니까, 봇들이 항상 오디언스부터 물음. 체크포인트 ①은 무조건 "경영진? 일반 직원?" 이었음. 이게 결과물 퀄리티를 끌어올린 **유일한 이유**일 수도 있음.

## 배운 점

### 💡 에이전트 수는 적을수록 좋다
처음 6개 → 최종 3개. 결과물 퀄리티는 오히려 올라감.
- 역할 경계가 명확해짐
- 동시에 말하는 사람이 적어져서 가독성 올라감
- "디자이너+이미지 생성 분리가 부자연스럽다" 판단이 결정적

### 💡 계획 단계에 공 들여야 실행이 빠르다
계획 3시간 vs 구현 30분. 바로 코딩부터 했으면 역전됐을 거.

### 💡 사람 편집자가 진짜 중요
소스 4개 주제 사건에서, 만약 100% 자동화였으면 "AI Agent 시장 패러다임 변화"만 만들고 나머지 3개는 잃어버렸을 거. **체크포인트 3개가 결정적 지점에 잘 배치된 게** 이번 실습의 핵심.

### 💡 오디언스·시사점 철학이 실제 결과물에 영향
경영진 대상이라 카피가 격식체 + 숫자 강조로 저절로 흘러감. 결과 슬라이드에 "우리 회사가 지금 뭘 해야 하는가"가 곳곳에 배어있음. 같은 소스로 "일반 직원" 오디언스였다면 완전 다른 카드뉴스 나왔을 듯.

### 💡 한국어 환경 = 항상 인코딩 조심
Git Bash on Windows + 한글 JSON body + UTF-8. bash 문자열 직접 넣지 말고 JSON 파일 경유.

### 💡 제목만 보고 판단 금지
소스 4개 주제 사건. 파일 제목으로 내용 짐작하지 말고 전체 읽기. 편집장 프롬프트에 "본문 전체를 먼저 읽고 흐름 파악한 후 분할" 원칙이 있었는데, 실제로는 안 지켜짐. 다음 버전엔 강제 검증 넣어야 함.
