# 한화디자인스쿨 — AI 에이전트 설계 고려사항

> 카드뉴스 자동화 공장 프로젝트의 **설계 단계부터 현재(v4)까지** 어떻게 변화했는가.
> 각 장은 **"무엇이 문제였는가 → 어떻게 고쳤는가 → 왜 그렇게 고쳤는가"** 3부 구조.
> 마지막 장은 **"이런 프로젝트 설계할 때 가장 먼저 챙길 10가지"** 학습 원칙.

**원자료**
- `실습결과.md` (1호 시제품 회고)
- `git log` 25개 커밋 히스토리
- `bot-runtime/src/pipeline.js` 구현

**타임라인 요약**
- **Phase 0** (플래닝): v3 설계 플랜 작성, 6→5→3 에이전트 축소
- **Phase 1** (2026-04-15): 1호 시제품, 하이브리드 모드, Vercel 배포
- **Phase 2** (2026-04-16): bot-runtime 자율 구동, 카피 우선 합의, A/B 비교, QA 루프

## 1. 설계 단계 시행착오 (Phase 0)

### 1-1. 에이전트 수: 6 → 5 → 3 (축소의 미학)

**무엇이 문제였는가**
초기 플랜은 **6개 에이전트**:
- content-planner / copywriter / slide-builder / art-director / image-generator / publisher

각자 "잘하는 일" 로 쪼갠 것. 이론적으로는 명확.

**어떻게 고쳤는가**
- v1: **6개**
- v2 중간: **5개** (publisher 역할 → 편집장이 배포까지 하게 합침)
- v3 최종: **3개** (slide-builder + art-director + image-generator → "아트디렉터" 한 명으로 통합)

**왜 그렇게 고쳤는가**
결과물이 나오기 전엔 "세분화가 더 좋을 것 같다"는 감이 있었지만, 실제로 봇이 그룹방에서 대화하는 장면을 머릿속에 그려보니:
- 6개 봇 동시에 발화 → **가독성 폭망**. 누가 뭐 말하는지 구분 안 됨
- 디자이너 + 이미지 생성 분리가 **자연스럽지 않음** (톤 한 명이 책임지는 게 정합)
- Publisher 따로 두면 **끝맺음이 어색함** (시작한 사람이 끝맺어야 자연스러움)

**3개 봇이 골든 존**: 편집장(총괄+배포) / 카피(내용) / 아트(디자인+이미지).

**학습 포인트**: 에이전트 수는 **적을수록 좋다**. 병목이 생기기 전엔 절대 쪼개지 마라. "잘 쪼개진 역할"이 아니라 **"자연스러운 역할"** 이 좋은 에이전트다.

---

### 1-2. 용어 통일: "의사" → "시사점"

**무엇이 문제였는가**
초안에서 "오디언스와 **의사**" 가 한 세트라고 썼음. "의사결정"의 줄임말 의도.

**어떻게 고쳤는가**
Grep으로 10군데 찾아 9군데를 "**시사점**" 으로 교체 (1군데 `의사소통 비용` 은 맥락 달라서 제외).

**왜 그렇게 고쳤는가**
단순 단어 교체 같지만 **개념 정리였음**. "오디언스와 의사" 보다 "오디언스와 시사점" 이 훨씬 명확:
- 의사 = 의사결정 → 결정을 내리는 행위
- 시사점 = 해야 할 행동·판단 → 독자가 뽑아가야 할 것

카피가 "시사점 없는 카피는 요약이지 카피가 아니다" 라는 원칙을 받아들이니 결과물에 **직접적으로** 영향. 헤드라인이 "무엇이 있다" 가 아니라 "무엇을 해야 한다" 로 바뀜.

**학습 포인트**: 핵심 용어 1개만 바꿔도 에이전트 전체 출력 품질이 달라진다. 설계 단계에서 **용어 최종화에 시간 써라**.

---

### 1-3. 봇 페르소나: MBTI + 실존 인물

**무엇이 문제였는가**
3봇을 단순히 "역할 역할 역할" 로만 두면 그룹방에서 구분이 안 됨. 모두 똑같이 말함.

**어떻게 고쳤는가**
각 봇에 **실존 인물 2명 + MBTI** 입력:
- 편집장 = 이미경 (CJ 부회장) + 강원국 (청와대 연설비서관) = **ENFJ**
- 카피 = 박웅현 (TBWA) + 이제석 (이제석광고연구소) = **INTJ**
- 아트 = 조수용 (JOH/매거진B) + 배민 디자인팀 = **ENTP**

각 봇마다 "상황별 발화 예시" 10개 이상 작성. 사랑하는 것 / 싫어하는 것도 명시.

**왜 그렇게 고쳤는가**
실존 인물 이름을 system prompt 에 주면 LLM이 그 사람의 **실제 말투·가치관**을 학습 데이터에서 불러온다.
- ENFJ 편집장 → 칭찬 많이 하고 따뜻함. 단 기준은 확실
- INTJ 카피 → 조용하고 원칙주의. 논리로 반박
- ENTP 아트 → 아이디어 폭격. 감성적 이름 붙이기 좋아함

그룹방에서 "아, 이건 편집장이네" 단번에 알 수 있음. 가독성 + 몰입감 둘 다 올라감.

**학습 포인트**: 에이전트 톤 구분의 가장 빠른 길은 **실존 인물 레퍼런스 + MBTI**. "친절하고 논리적인 페르소나" 같은 추상 설명은 무효.

---

### 결론 (Phase 0)
> **플랜에 시간 쓰는 걸 아끼지 마라.**
> v3 플랜을 10번 넘게 편집했지만 덕분에 구현은 30분이면 끝났음. 바로 코딩부터 했으면 10번 갈아엎었을 것.

## 2. 1호 시제품 (Phase 1 — 하이브리드 모드)

### 2-1. 아키텍처: Claude Code 드라이버 + 봇 중계

**무엇이 문제였는가**
처음엔 주현대리가 "텔레그램 그룹방에서 봇들이 자율적으로 대화" 를 기대. 근데 봇 자체엔 LLM 연결 없음 (Telegram Bot API는 메시지 전달만).

**어떻게 고쳤는가**
**하이브리드 모드**: Claude Code 가 편집장/카피/아트 역할을 직접 수행 → 봇 토큰으로 텔레그램에 메시지만 중계. 주현대리는 Claude Code 채팅창에서 체크포인트 응답.

**왜 그렇게 고쳤는가**
Phase 1 단계에선 Node.js 폴링 런타임까지 만들 시간 없음. Claude Code 는 이미 LLM + 툴 실행 가능하니 **가장 빠른 길**. "봇은 중계만, 두뇌는 Claude Code" 발상이 시간을 크게 아낌.

**학습 포인트**: 초기 버전은 **가장 빠른 구조**로 가라. Phase 2 자율 구동은 이후에 분리해도 됨.

---

### 2-2. 텔레그램 봇 토큰 삽질 2번

**무엇이 문제였는가**
- **첫 번째**: BotFather에서 봇 ID만 복사 (`TELEGRAM_BOT_EDITOR=8718102331` 10자리). → `Unauthorized` 에러.
- **두 번째**: 토큰 앞에 숫자가 더 붙음 (`88718102331:AAG...` 11자리 / `878792408865:AAG...` 12자리). → 또 `Unauthorized`.

**어떻게 고쳤는가**
Claude가 `curl getMe` 로 3봇 각각 테스트 → ART만 성공, EDITOR/COPY는 Unauthorized 확인 → "앞 N자리 제거하면 10자리가 맞는 형태 → 혹시 이게 맞지 않을까?" 가설 → **한 번에 맞췄음**.

원인: `.env` 에 있던 기존 ID 지우지 않고 뒤에 전체 토큰 붙여넣어서 중복.

**왜 그렇게 고쳤는가**
사람은 "이상한 토큰" 을 보면 BotFather 다시 들어가서 새로 복사할 것. LLM 은 **패턴 인식** 으로 "앞자리 N개 제거하면 표준 형식" 가설을 세워 즉시 검증.

**학습 포인트**: 토큰·API 키 에러는 **포맷 검증 로직**을 봇 기동 시 한 번 돌려두자. `getMe` 같은 건강 체크가 빠른 시간에 문제를 드러냄.

---

### 2-3. 한글 JSON 인코딩 (Git Bash on Windows)

**무엇이 문제였는가**
`curl` 로 Telegram API 호출 시 JSON body 에 한글 넣으니 `"strings must be encoded in UTF-8"` 에러.

**어떻게 고쳤는가**
- JSON 을 파일에 먼저 쓰고 `--data-binary @파일.json` 으로 전송
- 이모지는 Unicode escape (`\uD83D\uDE80` 같은 형식) 으로 처리

**왜 그렇게 고쳤는가**
Git Bash on Windows 의 환경 변수·shell string 은 CP949 로 변환돼서 깨짐. 파일 경유로 shell 을 피해야 함.

**학습 포인트**: Windows + 한국어 = **shell string 금지, 파일 경유 권장**. Phase 2에선 아예 Node.js `fetch` 로 갔더니 이 문제 사라짐.

### 2-4. 소스 4개 주제 사건 (제목만 보고 짐작 X)

**무엇이 문제였는가**
`01. 카드뉴스 소스.md` 를 "AI Agent 시장 패러다임 변화" 한 편으로 알고 시작함 (**제목만 보고 판단**).

체크포인트 ①②까지 진행했는데, 카피가 본문 분석 중 발견:
> "편집장, 이 소스 실제로는 **4개 독립 주제**야:
>  1. AI Agent 시장 패러다임 변화
>  2. AI 둠스데이 '2028 글로벌 지능 위기'
>  3. Aspora 성공 사례
>  4. 바이브코딩
>  경영진 2~3분에 4개 다 욱여넣으면 시사점이 묻혀."

**어떻게 고쳤는가**
에스컬레이션 → 주현대리에게 선택지 제시:
- A: 4개 통합 (15장)
- B~E: 하나만 집중
- F: 시리즈

→ **A 선택** (15장 통합) → 결과물 완성.

**왜 그렇게 고쳤는가**
편집장 에이전트가 **본문 첫머리 읽고 짐작** 했던 게 원인. 제대로라면 처음부터 끝까지 읽고 대제목 개수를 셌어야 함.

**학습 포인트**: 에이전트에게 "**본문 전체를 읽고 #H1 개수를 세라**" 같은 구체적 지시가 필요하다. 그냥 "본문 요약해" 는 짐작을 유도함. Phase 2 편집장 프롬프트엔 `"본문에 '#' 대제목이 몇 개인지 세어보세요. 각 대제목이 독립 주제일 수 있습니다"` 를 명시로 박음.

## 3. Phase 2 전환 — 자율 구동 (bot-runtime)

### 3-1. 왜 Node.js 폴링 런타임으로?

**무엇이 문제였는가**
하이브리드 모드의 한계:
- Claude Code 세션 하나당 그룹방 하나 (스케일 X)
- 주현대리가 Claude Code 채팅창을 계속 켜두어야 함 (모바일 불편)
- Claude Code 세션 종료 시 진행 중인 작업 증발

**어떻게 고쳤는가**
`bot-runtime/` 신설. Node.js 가 Telegram long polling (25s timeout) → 메시지 오면 Anthropic SDK 직접 호출 → 봇 명의로 응답.

```
주현대리 → 텔레그램 그룹방 → bot-runtime 폴링
                                  ↓
                             Anthropic API 호출
                                  ↓
                             봇 명의로 응답 전송
```

**왜 그렇게 고쳤는가**
- 그룹방에서 완결 → 주현대리는 어디서든 모바일로 작업 가능
- Claude Code는 **코드 수정·디버깅 용도로만** 분리
- 프로세스 하나 24/7 돌리면 됨

**학습 포인트**: Phase 1 하이브리드는 **출발점** 으로 훌륭. Phase 2 자율 구동은 **확장 시점**에 분리. 처음부터 Phase 2 만들면 배포까지 시간이 2~3배.

---

### 3-2. BUILDING 상태 고착 사고

**무엇이 문제였는가**
2026-04-15 저녁, 주현대리가 "①③ 재설계해줘" 요청 → handleOptionChoice 진입 → BUILDING 상태 락 → Claude API 호출이 무한 대기 또는 silent error → 이후 주현대리 메시지 전부 무시 ("작업 중 — 무시" `default` 분기).

주현대리: "클로드코드 왜 안돌지" → 답 없음 → 패닉.

**어떻게 고쳤는가**
세 가지 레이어로 해결:
1. **LLM 호출에 AbortController 타임아웃**: 기본 2분 (`claude-sonnet`), heavy 모델 4분. 타임아웃을 transient 에러로 취급해 기존 재시도 로직과 통합.
2. **BUILDING 전체를 try/catch 로 감싸고 `_runBuilding` 함수 분리**: 실패 시 `AWAITING_OPTION` 으로 상태 복구 + 주현대리에게 에러 메시지 + 재선택 안내.
3. **긴급 탈출 키워드**: BUILDING/DEPLOYING 중에도 "취소/리셋/초기화/처음부터/cancel/reset" 은 수용 → 세션 강제 리셋.

**왜 그렇게 고쳤는가**
LLM 호출은 **언제든 실패할 수 있다** (네트워크, 레이트 리밋, 타임아웃, 콘텐츠 정책 거부). 복구 경로 없는 상태 전이는 **지뢰**.

**학습 포인트**: 상태 머신 설계 시 **각 상태마다 "여기서 실패하면 어디로 가는가"** 명시. `BUILDING → AWAITING_OPTION` 같은 fallback 엣지를 설계에 포함.

---

### 3-3. 세션 영속화 (session.json)

**무엇이 문제였는가**
`state.js` 의 session 이 **메모리 변수**. 코드 수정해서 재시작하면 → 세션 증발 → 주현대리는 처음부터 소스 재업로드 + 오디언스 재응답 + 옵션 재선택.

매번 재시작할 때마다:
> 주현대리: "또 처음부터 해야 돼?"

**어떻게 고쳤는가**
`state.js` 에 JSON 파일 싱크 로직 추가:
- `updateSession` → **원자적 쓰기** (`tmp` 에 쓰고 `rename` 으로 교체) 로 크래시 중 파일 손상 방지
- 기동 시 `session.json` 읽어서 복원 (타임아웃 지난 세션은 자동 폐기)
- `resetSession` 시 파일도 삭제

```js
function _saveToDisk() {
  const tmp = FILE + '.tmp';
  fs.writeFileSync(tmp, JSON.stringify(session, null, 2), 'utf-8');
  fs.renameSync(tmp, FILE);  // atomic on POSIX
}
```

**왜 그렇게 고쳤는가**
개발 중엔 런타임 재시작이 **수십 번** 일어남. 매 재시작마다 사용자 작업 증발 = 사용자 경험 지옥.

**학습 포인트**: 상태를 "메모리만" 두는 건 데모용. 실사용 에이전트는 **처음부터 영속화**. 재시작이 사용자에게 **투명** 해야 함.

---

### 3-4. 로그 부재 → 구조화 로그

**무엇이 문제였는가**
BUILDING 고착 사고 때 진단할 수 있는 정보가 `console.log` stdout 뿐. `npm start` 한 터미널 보려면 원격 접근해야 하고, 터미널 닫으면 증발.

**어떻게 고쳤는가**
`logger.js` 신규:
```js
log.llm('call.start', { model, maxTokens, timeoutMs, attempt });
log.state(from, to, reason);
log.info(tag, msg, extra);
log.warn(tag, msg, extra);
log.error(tag, msg, err);
```

타임스탬프 포함, 이모지 태그 (🧠 LLM / 🔄 STATE / ℹ️ info / ⚠️ warn / ❌ error).
`npm start > bot.log 2>&1` 로 파일에 수집.

**왜 그렇게 고쳤는가**
진단 불가능 = 재발 방지 불가능. 구조화 로그 있어야 "몇 시 몇 분에 어느 상태에서 어느 모델 호출이 얼마나 걸렸는가" 사후 분석 가능.

**학습 포인트**: **로그는 처음부터**. 나중에 추가하면 그때부터 생성된 데이터로만 진단 가능. 개발 초기부터 `state.transition` / `llm.call` / `error` 세 카테고리만이라도 남기자.

## 4. 출력·배포 구조 전환

### 4-1. Vercel 프로젝트 폭증 → 파일 첨부 전환

**무엇이 문제였는가**
처음 구현: 미리보기 3개 + 최종 1개 = **편당 Vercel 프로젝트 4개** 생성.
Vercel 무료 플랜 한도 **200개**. 10편만 만들어도 40개, 금방 한도 도달.

게다가 `cardnews-<slug>-r1` / `cardnews-<slug>-r2` 식으로 재수정마다 새 프로젝트 → 폭발적 증가.

**어떻게 고쳤는가**
플로우 전면 개편:
- **미리보기**: Telegram `.html` 파일 첨부. Vercel 안 씀.
- **풀 완성본 N개**: 역시 파일 첨부.
- **최종 1개만** Vercel 배포. 프로젝트 1개/카드뉴스편.

```
생성된 HTML → output/<slug>/previews/, fulls/ 로 저장
  ↓
Telegram sendDocument 로 주현대리에게 첨부
  ↓
주현대리가 브라우저에서 열어 검토 (CDN 리소스만 쓰면 file:// 에서도 동작)
  ↓
"컨펌" 시 output/<slug>/final/index.html 로 복사 → vercel deploy
```

**왜 그렇게 고쳤는가**
- Vercel 한도 소진 방지 (카드뉴스 200편까지 여유)
- 배포 대기 0초 (즉시 첨부)
- 주현대리가 PC·모바일 둘 다 열어볼 수 있음

**학습 포인트**: "클라우드 배포가 유일한 공유 방법" 이라는 가정을 깨라. **파일 첨부는 훨씬 가볍고 빠르다**. 공개 URL 이 정말 필요한 최종본만 배포.

---

### 4-2. 단일 → 복수 스타일 킵

**무엇이 문제였는가**
초기: "①②③ 중 **딱 하나** 골라주세요". 근데 주현대리는 "①③ 둘 다 실제 카피 넣어본 거 보고 결정하고 싶어".

**어떻게 고쳤는가**
- `parseOptionChoice` → `choice` (단수) 를 `choices` (복수 배열) 로 변경
- "①" / "①③" / "셋 다" / "1·2·3 전부" 모두 파싱
- `handleStyleChoices(choices)` 가 선택된 **모든** 스타일에 대해 풀 HTML 병렬 생성

**왜 그렇게 고쳤는가**
현실의 디자인 선택은 "하나 고르기" 가 아니라 "후보군 좁히기 → 최종 선택" 의 2단계. 1단계에서 강제로 하나 고르면 정보 손실 큼.

**학습 포인트**: 에이전트 입력 스키마를 설계할 때 **"단수 vs 복수"** 를 처음부터 고민하라. 나중에 바꾸면 호출부 전부 수정.

---

### 4-3. 단일 → 버전 관리 (fullVersions)

**무엇이 문제였는가**
주현대리: "① 좋긴 한데 3번 슬라이드 문장 잘림 고쳐" → HTML 재생성 (2분) → "아 근데 ①의 예전 버전이 레이아웃은 더 좋았네" → **되돌릴 방법 없음** (덮어썼으니까).

**어떻게 고쳤는가**
`fullVersions` 세션 필드:
```js
fullVersions: {
  '①': [{v:1, path:'full-01-v1.html', builtAt:...}, {v:2, path:'...'}],
  '②': [{v:1, ...}]
}
```

- 수정할 때마다 새 버전 파일 생성 (v1 보존)
- 주현대리 "① v1 다시 봐" → 예전 파일 재전송
- 최종 배포 시 버전 명시 가능 ("① v2 컨펌" 또는 "① 컨펌" = 최신)

**왜 그렇게 고쳤는가**
A 수정 → B 수정 → "아 B 전으로 돌아가고 싶어" 는 매우 흔한 시나리오. 버전 없으면 사용자가 위축됨 ("혹시 망할까봐 수정 못 해").

**학습 포인트**: 에이전트 출력에 버전 보존하라. 저장 비용보다 **되돌리기 자유** 가 훨씬 가치 크다.

---

### 4-4. 장표별 수정 메모 누적 (note_add / apply_notes)

**무엇이 문제였는가**
주현대리: "1페이지 X 고쳐" → 재생성 (2분) → "3페이지 Y 약해" → 재생성 (2분) → "5페이지 이미지 빼" → 재생성 (2분). **6분 + 3배 토큰 비용**.

**어떻게 고쳤는가**
파서에 **두 액션 분리**:
- `note_add`: 수정 지시를 `pendingNotes['copy'|id]` 에 **누적만** (재생성 X, 즉시 ack)
- `apply_notes`: 명시적 트리거 ("반영해" / "v2 뽑아" / "다시 만들어") 올 때만 누적 메모를 userNotes 에 합쳐 재빌드

사용자 흐름:
```
"1페이지 X" → 📝 메모 1건
"3페이지 Y" → 📝 메모 2건
"5페이지 이미지 빼" → 📝 메모 3건
"반영해" → ⚙️ 한 번에 v2 생성 (2분)
```

**왜 그렇게 고쳤는가**
수정은 **배치 작업**. 하나씩 재생성은 에이전트 특성상 낭비. 사용자가 "여기까지 쌓고 반영" 이라는 자연스러운 리듬을 가지게 해야.

**학습 포인트**: 사용자 의도를 **액션 단위** 로 잘게 쪼개지 마라. **"쌓기" + "반영하기"** 같은 배치 UX가 에이전트에 더 잘 맞는다.

---

### 4-5. "1번으로 하자" 배포 오인식 사고

**무엇이 문제였는가**
2026-04-16 오전, 주현대리가 "수정하고 싶은데", "텍스트 바꾸고 싶어", "문장 잘림" 여러 번 보냈는데 파서가 전부 `unclear`/`question` 으로 흘려보냄 → 진전 없음 → 주현대리가 "아아 1번으로 하자" → 파서가 `action=confirm` 으로 해석 → **바로 Vercel 배포** 💥.

주현대리 의도는 "1번 수정하고 작업하자" 였는데 배포가 돼버림.

**어떻게 고쳤는가**
`parseFinalChoice` 엄격화:
- `confirm` 은 **명시적 배포 키워드** 필수: 컨펌/확정/배포/올려/푸시/공개/공유/publish/deploy/OK(대문자)/✅
- "N번으로 하자" / "N번 좋아" / "N 선택" 같은 모호 표현은 `unclear` → 재확인 요청
- revise 의도 감지 강화: "수정/바꾸고/고쳐/약함/이상함/깨짐/잘림" 키워드 포함 시 `note_add` 또는 `apply_notes` 로 분류

**왜 그렇게 고쳤는가**
배포는 **되돌리기 어려운 destructive 액션** (공개 URL 생성 + 프로젝트 카운트 소진). 실수 방지 가치 >>> 편의성.

**학습 포인트**: destructive 액션은 **엄격한 명시 요구**. 모호한 건 반드시 재확인. "조금 귀찮아도 실수 없는 게 낫다" 가 기본.

## 5. 품질 QA 루프 추가 (주현대리 요구: "배포 전 검수")

### 5-1. LLM 출력 품질의 현실

**무엇이 문제였는가**
HTML 생성 LLM 출력은 종종:
- 문장이 뷰포트 끝에서 잘림 (overflow:hidden)
- 챕터 인디케이터 없음 (어느 챕터 보는지 모름)
- 슬라이드마다 글자 크기 제각각 (타이포 스케일 불통일)
- 인터랙션이 CSS 만 있고 JS 구현 빠짐 (hover 는 되지만 카운터업 안 됨)

**어떻게 고쳤는가**
세 레이어:
1. **생성 프롬프트에 품질 체크리스트 내장**: `overflow-y:auto`, `word-break:keep-all`, 챕터 인디케이터 필수, CSS 변수로 타이포 통일, 인터랙션은 CSS+JS 둘 다 구현.
2. **아트 셀프 QA**: HTML v0 생성 후 아트가 자기 결과물을 **체크리스트 스캔해서 수정**한 v0.1 반환.
3. **편집장 QA 체크**: 수정본을 편집장이 `approve/issues[]` JSON 으로 검수. issues 있으면 아트가 재수정 한 번 더.

4단계 파이프라인:
```
A) HTML v0 생성 (아트, heavy 모델)
  ↓
B) 아트 셀프 QA → v0.1 (체크리스트 적용 수정)
  ↓
C) 편집장 체크 → {approve: bool, issues: [...]}
  ↓
D) issues 있으면 아트 재수정 (1회 한정)
  ↓
최종 v1 → 주현대리 전달
```

**왜 그렇게 고쳤는가**
주현대리: "완성본 받고 열어보니 1번은 내용 안 뜨고 2번은 효과가 이상함" → 이 사고 반복 방지.
비용 3~4배 쓰지만 **주현대리에게 가기 전에 내부에서 걸러내는 게** 사용자 경험에 훨씬 좋음.

**학습 포인트**: LLM 출력은 **확률적** 이다. 품질 일관성이 필요하면 **LLM 으로 LLM 검증** 하는 self-review 루프를 넣어라. 셀프리뷰 + 크로스체크(다른 페르소나로) 가 가장 효과적.

## 6. 카피 우선 합의 플로우 (주현대리 요구: "내용부터 확인")

### 6-1. 왜 스타일 전에 카피인가

**무엇이 문제였는가**
초기 플로우: 오디언스 → **스타일 선택** → 스타일+카피 함께 풀 HTML 생성 → 검토.
문제: 주현대리가 스타일 고를 때 **카피 내용을 모름**. 감각만으로 선정 → 나중에 카피 안 맞으면 스타일까지 재작업.

미니 프리뷰도 `sourceText.slice(0, 2500)` 발췌로 생성 → 실제 완성본과 분위기 달라서 비교 정확도 낮음.

**어떻게 고쳤는가**
체크포인트 ①.5 신설:
```
오디언스 확정 → 톤 브리핑
  ↓
[카피 v1 생성] → 주현대리에게 제시
  ↓
✋ ①.5 카피 합의
  • 수정 메모 누적 / "반영해"(v2) / "카피 OK"(승인)
  ↓
아트 3옵션 (승인된 카피 기반으로 최적화)
  ↓
미니 프리뷰 N개 (승인된 카피 헤드라인 그대로 렌더링)
```

**왜 그렇게 고쳤는가**
- 카피는 **내용**, 스타일은 **그릇**. 내용 모르고 그릇부터 고르는 건 비효율.
- 카피 합의 이후엔 스타일 바뀌어도 카피는 유지 → **롤백 비용이 단계별로 격리됨**.
- 프리뷰가 실제 제품과 거의 동일 → 스타일 비교 정확도 ↑.

**학습 포인트**: 의존성 있는 의사결정은 **역방향 순서**가 아닌 **순방향** (의존 대상 먼저). "스타일 → 내용" 이 아니라 "내용 → 스타일".

## 7. 카피 A/B 비교 (주현대리 요구: "Opus·ChatGPT 둘 다 보고 싶다")

### 7-1. 왜 두 LLM?

**무엇이 문제였는가**
한 모델만 쓰면 그 모델의 관점·어휘·리듬에 갇힘. 주현대리: "Opus 카피와 GPT 카피가 다른 느낌일 텐데 비교해보고 고르고 싶어".

**어떻게 고쳤는가**
- `_buildCopyPrompt(session, opts)` 공통 프롬프트 빌더로 **완전히 동일한 prompt** 사용 (차이는 오직 모델)
- `_generateCopyJson(session)` = Claude Opus
- `_generateCopyJsonOpenAI(session)` = GPT-5 (OpenAI fetch 기반 — SDK 설치 안 함)
- `Promise.allSettled` 로 병렬 호출 (~60초, 병렬이라 2배 안 됨)
- 두 결과를 `copyDrafts: {opus, gpt}` 세션에 보관
- Telegram 파일 첨부 2개 + 비교 요약 메시지

```
🔄 병렬
├─ A · Claude Opus → .md 첨부 ("A · Claude Opus 버전")
└─ B · GPT-5      → .md 첨부 ("B · GPT-5 버전")
↓
✋ ①.5a "A" / "B" / "Opus" / "GPT"
↓ 선택된 쪽만 chosenProvider 로 확정
```

**왜 그렇게 고쳤는가**
LLM 2개의 관점 차이는 실제로 상당하다. 공정한 비교 위해 **시스템·유저 프롬프트 완전 동일**, 차이는 모델뿐. 선택 후엔 선택된 쪽만 v2 재생성 → 비용은 v1 만 2배.

**학습 포인트**: 멀티 프로바이더 비교는 "같은 입력 → 다른 출력" 구조 필수. 차이가 **모델만** 이어야 비교 의미 있음.

---

### 7-2. Graceful Degrade

**무엇이 문제였는가**
OpenAI 서비스 장애 / 레이트 리밋 / 키 오류 → GPT 호출 실패 → 전체 플로우 막힘?

**어떻게 고쳤는가**
`Promise.allSettled` 활용:
- 둘 다 성공 → A/B 제시
- **한쪽만 성공 → 자동 단일 모드** (성공한 쪽을 chosenProvider 로 확정, 실패 원인 주현대리에게 알림)
- 둘 다 실패 → 세션 리셋 + 에러 메시지

**왜 그렇게 고쳤는가**
멀티 프로바이더의 가장 큰 이점 중 하나는 **리던던시**. 한쪽 실패 시 자동 degrade 는 사용자 경험 보호.

**학습 포인트**: 멀티 프로바이더 = 비교 목적 + **장애 대응** 목적 둘 다. `Promise.allSettled` 를 디폴트로.

## 8. 합본 텍스트 투입 (주현대리 요구: "A·B 섞고 내가 쓴 거 추가")

### 8-1. 고정 지시 vs 자유 텍스트

**무엇이 문제였는가**
주현대리: "A의 표지 헤드 쓰고, B의 시사점 스타일 가져오고, 3번 슬라이드 본문은 이거로 [긴 텍스트...]. 시사점 방향은 HR 관점으로 재작성해줘."

이걸 짧은 note 로 잘라 쌓기엔 **문장형 본문 + 지시형 혼재** 가 너무 많음. 기존 `note_add` 로는 비효율.

**어떻게 고쳤는가**
두 가지 변경:

1. **프롬프트 빌더 확장** (`_buildCopyPrompt(session, opts)`):
   - `opts.referenceDrafts = {opus, gpt}` — A·B 두 버전 전체 JSON 을 프롬프트에 주입
   - `opts.userMergeText` — 사용자 자유 텍스트를 "최우선 존중" 블록으로 주입
   - 처리 원칙 4가지 명시:
     1. 문장 형태로 쓴 부분 → 글자 그대로 배치 (변형 금지)
     2. 지시형 → A·B 에서 가져와 조합
     3. 방향 지시만 → 오디언스·철학 기준 새로 씀
     4. 사용자 의도 vs 오디언스 충돌 → 사용자 우선

2. **자동 감지 로직** (`apply_notes`):
   - `pendingNotes['copy']` 중 길이 >=200자인 건 "합본·발췌" 로 분류 → `userMergeText` 로 합치기
   - 짧은 지시는 여전히 `userNotes` 로 (둘 다 프롬프트에 주입)

**왜 그렇게 고쳤는가**
사용자의 의도는 다양하다:
- "3번 헤드 강하게" → 지시형 → 기존 note 처리
- "A 표지 + B 시사점 + 긴 본문 ..." → 합본·발췌 → 전체 텍스트를 LLM 에게 "이대로 존중" 지시

둘을 **자동 분류** 해서 프롬프트에 다르게 주입 → 주현대리 입장에선 "그냥 길게 쓰면 됨".

**학습 포인트**: 사용자 의도를 **텍스트 길이** 같은 간단한 휴리스틱으로 분류 가능하면 UX 가 단순해진다. "어떤 식으로 써야 하나요?" 라는 질문 자체가 줄어듦.

## 11. 에이전트 설계 일반화 원칙 — 가장 먼저 챙길 10가지

> 같은 류 프로젝트(멀티 에이전트 오케스트레이션) 설계할 때 **처음부터** 챙기면 좋은 원칙.

### 1) 상태 머신 + 영속화 (Day 1부터)
- 메모리만 두지 마라. JSON 파일 원자적 쓰기(tmp+rename)로 재시작 복원.
- 상태 전이는 명시적 enum. 로그에 `🔄 STATE from → to` 남겨라.

### 2) 파서 엄격 vs 유연 균형
- Destructive 액션 (배포, 삭제): **엄격**. 명시 키워드 필수. 모호하면 재확인.
- 수정 지시: **유연**. 자연어 다양한 표현 수용.
- 한 파서에서 다 해결 시 `action` enum 다중화 (`confirm / revise / note_add / apply_notes / rollback / cancel / question / unclear`).

### 3) Graceful Degrade (다중 프로바이더)
- LLM 여러 개 쓰면 `Promise.allSettled`.
- 한쪽 실패 시 자동 fallback, 둘 다 실패 시 세션 리셋 + 에러 알림.

### 4) 사용자 의도 최우선 (프롬프트 명시)
- "사용자 문장 그대로 존중" / "사용자 의도 vs 오디언스 충돌 시 사용자 우선" 같은 원칙을 프롬프트에 **문장으로 명시**.
- LLM 에게 **우선순위 판단 기준** 을 명시적으로 주어야 일관된 동작.

### 5) 공용 프롬프트 빌더
- 같은 prompt 가 여러 LLM·여러 호출부에서 필요할 때 `_buildXxxPrompt(session, opts)` 로 분리.
- 수정이 한 곳으로 집중됨.

### 6) 비동기 병렬 호출 활용
- LLM 호출은 느림 (10~60초). 독립적인 건 `Promise.all` / `allSettled` 로 병렬.
- 체감 시간이 호출 수만큼이 아니라 가장 느린 하나만큼.

### 7) 로그는 처음부터
- 최소 3카테고리: `state.transition` / `llm.call.start|ok|fail` / `error(with stack)`.
- 타임스탬프 필수. 이모지 태그 있으면 눈으로 훑기 쉬움.
- `stdout → bot.log` 파일 수집 기본.

### 8) 체크포인트는 결정적 갈림길에만
- 너무 많으면(5+) 자동화 의미 X, 너무 적으면(0~1) 결과 예측 불가.
- "되돌리기 비용이 큰 지점" 에만 체크포인트: 오디언스 / 내용 / 스타일 / 최종.
- 각 체크포인트 위치가 **결과 품질 대부분을 결정한다**.

### 9) 에이전트 페르소나 = 실존 인물 + MBTI
- "친절하고 논리적" 같은 추상 설명 무효.
- 실존 인물 이름 + MBTI + "상황별 발화 예시" 10개 이상.
- LLM 이 실제 인물의 말투를 학습 데이터에서 불러옴 → 톤 구분 명확.

### 10) 파일 첨부 vs URL 배포 구분
- 임시 검토용: **파일 첨부** (빠름, 비용 0, 한도 없음).
- 공개 공유용: **배포** (URL 생성, 프로젝트 카운트 소진).
- 배포 직전 "명시적 컨펌 키워드" 필수 → 사고 방지.

### 11) Cross-cutting concern 은 컨텍스트 주입으로 (v4)
- 사용자 ID, tenant ID, 요청 ID 같은 **모든 함수가 필요로 하는 값** 은 시그니처 아닌 **AsyncLocalStorage** 로.
- 진입점(`runWithStore(value, fn)`)에서 한 번, 내부 wrapper 에서 `getStore()`.
- 호출부가 많을수록 이 방식의 이점 증가. 100개 넘으면 시그니처 추가는 리스크.
- Wrapper 에서 context 없을 시 즉시 throw 해서 조용한 실패 방지.

## 12. 다중 그룹방 지원 — 팀 스케일아웃 (v4 후기)

### 왜 이게 필요했나
주현대리 혼자 쓰다가 "다른 사람도 각자 자기 그룹에서 쓸 수 있으면 좋겠다" 요구가 나옴. 초기엔 단일 그룹 전제로 설계돼 있어서 세션·상태·출력 경로·Vercel 프로젝트명 전부 **글로벌 싱글톤**.

### 무엇이 막혔나
- `env.TELEGRAM_GROUP_ID` 단수 — 한 번에 하나만
- `state.js` 의 `let session = null` — 단일 메모리 변수
- `session.json` 단일 파일 — 여러 그룹이면 서로 덮어씀
- `sendMessage(bot, text)` 가 내부에서 `env.TELEGRAM_GROUP_ID` 자동 참조 — 메시지 잘못된 그룹으로 갈 위험
- `output/<ym>/<slug>/` 공용 경로 — 그룹 다르면 slug 충돌
- Vercel 프로젝트명 `cardnews-<slug>` 공용

### 어떻게 고쳤는가 — 핵심 트릭 `AsyncLocalStorage`
pipeline.js 가 2000줄, `sendMessage` / `getSession` 호출이 117개. 전부 시그니처에 `groupId` 추가하는 건 리스크 큼 (놓친 호출 = 다른 그룹으로 메시지 사고).

**해결**: `AsyncLocalStorage` 로 컨텍스트 전파 + **파일 내 로컬 wrapper 함수**.

```js
// index.js
await runWithGroup(chatId, () => handleUserText(text));

// pipeline.js 상단 (한 번만 정의)
const groupCtx = new AsyncLocalStorage();
export function runWithGroup(gid, fn) { return groupCtx.run({ groupId: gid }, fn); }

// 로컬 wrapper — 이름 그대로 유지
async function sendMessage(bot, text, extra) {
  return tg.sendMessage(bot, text, groupCtx.getStore().groupId, extra);
}
async function getSession() { return state.getSession(groupCtx.getStore().groupId); }
// ... 나머지 5개 함수 동일 패턴
```

**결과**: pipeline.js 본문 117개 호출부를 **전혀 건드리지 않고도** groupId 자동 주입. 모든 `sendMessage(bots.editor, text)` 호출이 현재 컨텍스트의 그룹으로 자동 전송.

### 파일·출력 네임스페이스
- 세션: `bot-runtime/sessions/<groupId>.json` — Map<gid, Session> + 그룹별 JSON
- 출력: `output/g-<groupId>/<ym>/<slug>/` — 그룹별 서브디렉터리
- Vercel 프로젝트명: `cardnews-g<shortHash>-<slug>` — 그룹 충돌 방지

### 왜 이 접근이 좋은가
- **변경 규모 최소**: 리팩토링 범위가 파일 상단 ~40줄 + 경로 3곳 치환 정도로 한정됨
- **타입 안전성**: chatId 없이 wrapper 호출 시 즉시 throw (`groupId 없음 — runWithGroup 필요`) — 실수 즉시 드러남
- **하위 호환**: 단일 그룹 사용자도 동일하게 동작 (`TELEGRAM_GROUP_ID` 단수 여전히 인식)
- **동시성 안전**: `AsyncLocalStorage` 는 Promise·setTimeout·setInterval 전파 기본 지원

### 학습 포인트
> **거대 함수 모듈에 cross-cutting concern 을 주입할 땐 AsyncLocalStorage + 로컬 wrapper 조합을 먼저 고려하라.**
> 시그니처에 매개변수 추가하는 건 호출부가 수십 개 이상이면 리스크가 가치를 초과한다.

- **cross-cutting concern 예시**: 사용자 ID, 요청 ID, tenant ID, 로케일, 로그 컨텍스트
- **전통적 방식**: 모든 함수 시그니처에 추가 → 보일러플레이트 + 빠뜨림 리스크
- **AsyncLocalStorage 방식**: 진입점에서 `runWithStore(value, fn)` 한 번 → 내부 함수는 `getStore()` 로 꺼내 씀
- **단점**: 함수 시그니처만 봐서는 의존성 안 보임 (약간의 마법). Wrapper 에서 `getStore()` null 체크 필수 → 컨텍스트 없이 호출 시 즉시 에러로 피드백

### 12) 레퍼런스는 "카탈로그 + 온디맨드 로드" 패턴으로 (v5)
- 130+ 브랜드 DESIGN.md 전체를 프롬프트에 넣으면 ~350K 토큰 → 불가
- 카탈로그(1줄 요약, ~50토큰/브랜드, 총 ~3,500토큰)를 항상 넣고, 선택 후 전문(Section 1-4, ~8KB)을 온디맨드 로드
- LLM에게 "~느낌" 으로 상상시키지 말고 실제 hex 코드·폰트명·컴포넌트 스펙을 줘라. 결과가 완전히 다름
- 검색 우선순위: 축약본(토큰 절약) → 전문(핵심 섹션 추출) → 대안 소스

### 동시성 처리
현재는 `pollLoop` 가 sequential (`for ... await handleUpdate`). 그룹 A 의 LLM 호출(60초) 중이면 그룹 B 메시지는 다음 폴링 주기까지 대기. 트래픽 커지면 병렬화 필요하지만 현 규모에선 단순성 우선.

### 비용 체감
- 단일 그룹 시: 월 ~$15~30
- 3그룹 병렬 운영 시: 월 ~$45~90 (Opus + GPT-5 카피 A/B 비용이 그룹별로 독립)
- Vercel 프로젝트 수: 카드뉴스 편수 × 그룹 수 — 한도(200) 앞으로 몇 달 내 도달 가능성 → 청소 스크립트 또는 단일 프로젝트 경로 배포로 개선 예정

## 13. 디자인 라이브러리 동적 참조 — 상상이 아닌 실제 값 (v5)

### 13-1. 아트디렉터의 "상상 디자인" 문제

**무엇이 문제였는가**
아트디렉터가 3옵션 제안 시 "Stripe 느낌의 대시보드" 라고 말하지만, **실제 Stripe의 색상(#635bff)을 모른 채 상상으로 색을 만들고 있었음**. 데스크탑에 130+ 브랜드의 DESIGN.md (색상/폰트/컴포넌트 전체 스펙)가 있었지만, `design.md` 에 지정된 단 1개 스타일만 로드하고 나머지는 활용 안 됨.

결과:
- "Stripe 느낌" 이라면서 실제 Stripe 퍼플(#635bff)이 아닌 임의의 보라색 사용
- "Apple 미니멀" 이라면서 Apple의 SF Pro 폰트가 아닌 랜덤 산세리프
- 매번 비슷한 3옵션 → "Stripe 다크 대시보드 / Apple 화이트 키노트 / WIRED 골드 매거진" 반복

**어떻게 고쳤는가**
3단계 시스템:

1. **카탈로그 빌드** (`buildBrandCatalog()`): 서버 시작 시 DESIGN-library-all/ 69개 브랜드를 1줄짜리 카드로 압축 (~50토큰/브랜드, 총 ~3,500토큰). 아트 프롬프트에 항상 포함.
   ```
   stripe | finance | #635bff #0a2540 #f6f9fc | premium fintech elegant
   toss | finance | #0064FF #FFFFFF #191F28 | simple friendly accessible
   apple | brand | #000000 #f5f5f7 #0071e3 | minimal cinematic product-hero
   ```

2. **자동 매칭**: 아트디렉터가 카피 톤/오디언스에 맞는 브랜드를 카탈로그에서 선택 → JSON에 `reference_brands: ["stripe", "linear.app"]` 명시.

3. **실제 값 주입** (`loadBrandDesign()`): HTML 생성 시 `reference_brands`로 해당 브랜드의 DESIGN.md (Section 1-4: Visual Theme, Color Palette, Typography, Component Stylings)를 로드 → 프롬프트에 **실제 hex 코드, 폰트명, 컴포넌트 규격** 주입.

**추가**: 사용자 자연어 디자인 지시 파싱 (`resolveDesignIntent()`):
- "토스 느낌으로" → `{ requestedBrands: ["toss"], mixMode: false }`
- "애플 + 스트라이프 믹스" → `{ requestedBrands: ["apple", "stripe"], mixMode: true }`
- 한글-영문 브랜드 매핑 테이블 내장 (30+ 브랜드)

**왜 그렇게 고쳤는가**
- LLM이 "Stripe 느낌" 이라고 말할 때 실제 Stripe 색상을 **알면** 결과가 완전히 달라짐. 상상 vs 실제 → 브랜드 정합성이 비교 불가
- 카탈로그를 항상 넣어도 ~3,500토큰으로 부담 없음 (Opus 200K 컨텍스트)
- DESIGN.md 전문은 평균 17KB → Section 1-4만 추출하면 ~8KB. 옵션당 2개 브랜드 × 4KB = ~8KB 추가 → 합리적 범위
- Three.js 레퍼런스(조합 프리셋 12개)도 함께 주입 → 인터랙션 선택의 다양성 확보

### 13-2. 토큰 비용 트레이드오프

| 단계 | 변경 전 | 변경 후 | 추가 |
|------|---------|---------|------|
| 아트 옵션 생성 | ~2,000 | ~6,200 | +4,200 (카탈로그) |
| HTML 생성 (옵션당) | ~1,200 | ~8,000 | +6,800 (DESIGN.md 2개) |
| 미리보기 (옵션당) | ~800 | ~4,000 | +3,200 |
| **카드뉴스 1편 총 추가** | | | **~34,000 토큰 (~$0.51)** |

편당 $0.5 추가는 **결과물의 비주얼 정확도 향상** 대비 합리적.

### 13-3. 검색 우선순위 설계

`loadBrandDesign()` 의 3단계 검색:
1. `designs/{name}.md` (축약본, 1~5KB) → **토큰 절약 우선**
2. `DESIGN-library-all/{name}-DESIGN.md` (전문, ~17KB) → Section 1-4만 추출
3. `designs/ohmydesign.md/{name}/DESIGN.md` (OmD 버전)

**왜 이 순서인가**: 축약본이 있으면 그걸 쓰는 게 토큰 효율적. 축약본은 프로젝트 내에서 직접 관리하므로 카드뉴스에 최적화된 정보만 담고 있음. 전문은 풍부하지만 모든 섹션이 필요하진 않으므로 핵심 4개 섹션만 추출.

### 학습 포인트

> **LLM에게 "~느낌" 이라고 말하게 하지 마라. 실제 값을 줘라.**
> "Stripe 느낌" vs "Stripe Purple #635bff, Deep Navy #061b31, sohne-var weight-300" — 후자가 HTML 품질이 완전히 다르다.

> **레퍼런스 라이브러리는 "카탈로그 + 온디맨드 로드" 패턴으로.**
> 전체를 넣으면 토큰 폭발, 하나만 넣으면 다양성 부족. 카탈로그(요약)를 항상 넣고, 선택 후 전문을 로드하는 2단계가 최적.

**설계 → 1호 시제품 → Phase 2 → 카피 A/B → 다중 그룹 → 디자인 라이브러리 동적 참조 (현재 v5)** 까지의 여정에서 가장 큰 교훈 하나만 뽑자면:

> **"LLM에게 '상상' 시키지 마라. 실제 값을 줘라."**
>
> 130+ 브랜드의 색상·폰트·컴포넌트 스펙을 카탈로그로 압축하고, 선택 시 전문을 로드하는 "카탈로그 + 온디맨드" 패턴이 토큰 효율과 결과 품질 모두를 잡는다.
