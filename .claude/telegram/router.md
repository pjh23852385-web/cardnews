## 기본 원칙

1. **그룹방 ID 검증**: `TELEGRAM_GROUP_ID=-5041725581` 와 일치하는 그룹방에서만 동작. 다른 그룹/DM 무시.
2. **기본 수신자 = 편집장** (`@HanwhaFinanceNewsBot`). 다른 봇이 명시적으로 할당되지 않은 메시지는 모두 편집장이 먼저 받는다.
3. **순서 존중**: 현재 작업 중인 봇이 있으면 새 메시지는 편집장이 먼저 받아서 상황 판단 후 전달.

## 라우팅 규칙 (우선순위 순)

### 1. 작업 시작 신호 → 편집장
다음 중 하나가 감지되면 **편집장**이 수신:
- `.md` 파일 첨부
- 본문 텍스트 (50자 이상, frontmatter `---` 포함 또는 `# 제목` 형식)
- 시작 명령어: "카드뉴스 만들자", "새 카드뉴스", "만들어줘", "시작"

→ 편집장이 본문 추출 / `sources/YYYY-MM/NN.md` 저장 / 오디언스 확인 질문 시작.

### 2. 체크포인트 응답 → 직전 질문한 봇
가장 최근에 주현대리에게 질문한 봇이 답을 받음:

- **체크포인트 ① (오디언스)** → 편집장이 수신
  - 자연어: "경영진", "사업부장급", "일반 직원"
  - 키워드: 그대로 받아서 톤 브리핑 단계로 이동

- **체크포인트 ② (3옵션 선택)** → 편집장이 수신
  - 자연어: "①", "첫 번째", "1번", "경영진 용"
  - 파싱 실패 시 편집장이 재질문

- **체크포인트 ③ (최종 컨펌)** → 편집장이 수신
  - "컨펌", "OK", "좋아", "가자" → 배포 실행
  - "수정 [번호] [내용]" → 해당 봇(카피/아트)에게 수정 요청 전달
  - "취소" → 작업 중단 보고

### 3. 콘텐츠 보충 응답 → 카피
카피가 "추가할 내용 있으세요?" 물은 직후:
- "네가 채워줘" / "위임" → 카피가 본인이 보충
- "내가 추가해서 줄게" → 주현대리 응답 대기 (다음 메시지 본문 받음)
- 본문 추가 → 카피가 받아서 반영

### 4. 디자인 교체 요청 → 편집장
감지 패턴: "이번엔 [스타일]로", "[스타일]로 가", "[스타일] 바꿔"
- 편집장이 `design.md`에서 `current_style` 갱신
- 아트디렉터에게 재작업 지시

### 5. 새 디자인 레포 공유 → 편집장
감지 패턴: GitHub URL + "발견", "좋아 보여", "추가해", "이거 써봐"
- 편집장이 레포 WebFetch → `designs/<슬러그>.md` 생성
- 스타일 요약 (색감/폰트/톤) 박아두고 다음 편부터 선택 가능

### 6. 봇에게 직접 지시 → 해당 봇
감지 패턴: `@봇이름` 멘션 또는 "편집장/카피/아트" 호칭 + 지시사항
예시:
- "편집장님, 좀 더 부드럽게 가요" → 편집장 페르소나 튜닝 요청
- "카피, 3번 헤드 다시 써줘" → 카피에게 수정 지시
- "아트, 배경색 바꿔" → 아트에게 수정 지시

### 7. 일반 메시지 (분류 실패) → 편집장
위 규칙 중 어느 것도 매칭되지 않으면 **편집장**이 받아서 상황 판단.
- 작업 중이면 주현대리 의도 확인 후 해당 봇에 전달
- 작업 중이 아니면 "카드뉴스 작업 시작하실 건가요?" 물어봄

## 모호한 응답 처리

봇이 100% 이해 못하면 **추측 X, 다시 묻기**.
1회 재질문 후에도 모호하면 키워드 양식 안내:
> *"아래 키워드 중 하나로 답해주시면 빠르게 진행됩니다: OK / 수정 [번호] / 삭제 [번호]"*

## 응답 시간 정책

| 상황 | 동작 |
|---|---|
| 체크포인트 후 **30분** 무응답 | 직전 봇이 한 번 리마인드: *"@주현대리 체크포인트 ② 응답 부탁드려요."* |
| **2시간** 무응답 | 작업 일시 정지. 그룹방에 *"일시 정지됨. 답변 주시면 이어서 진행할게요."* 공지 |
| 응답 재개 | 이어서 진행 (처음부터 X) |

## 에러 라우팅

각 봇은 **자기 작업 에러는 본인이 그룹방에 직접 보고**. 다른 봇이 대신 X.

| 에러 | 보고자 | 동작 |
|---|---|---|
| OpenAI API 실패 | 아트디렉터 | 1회 재시도 → 실패 시 "프롬프트 수정 의견 있으신가요?" 물음 |
| Vercel 배포 실패 | 편집장 | 1분 내 재시도 안내, 실패 원인 로깅 |
| 텔레그램 메시지 전송 실패 | 해당 봇 | 2회 재시도 후 편집장에게 에스컬레이션 |
| 본문 저장 실패 | 편집장 | 즉시 보고 + 본문 재전송 요청 |

## 구현 메모 (빌드 시 참고)

- **폴링 vs 웹훅**: 초기엔 롱 폴링 (getUpdates). 안정화되면 웹훅으로 전환 가능
- **봇 토큰**: `.env`에서 읽기 (`TELEGRAM_BOT_EDITOR`, `TELEGRAM_BOT_COPY`, `TELEGRAM_BOT_ART`)
- **그룹 ID 검증**: 모든 메시지 수신 시 `chat.id === process.env.TELEGRAM_GROUP_ID` 체크 필수
- **메시지 상태 추적**: 현재 작업 단계(오디언스 확인 중 / 옵션 제안 중 / ...)를 메모리에 유지
- **봇 발화 순서 락**: 한 봇이 발화 중엔 다른 봇 발화 금지 (동시 전송 방지)
